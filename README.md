# Архитектурные решения
### Почему Wallet вынесен в отдельную сущность?
Я отказался от хранения баланса напрямую в таблице users (плоская структура) в пользу связи 1:1 (User -> Wallet) по следующим причинам:

- Разделение доменов (Domain Isolation):
Таблица users отвечает за идентификацию и профиль (логин, почта, хеш пароля). Таблица wallets отвечает за финансовое состояние. Изменение логики профиля не должно аффектить финансовый движок и наоборот.

- Блокировки и Конкурентность (Locking Strategy):
При выполнении транзакции мы используем SELECT FOR UPDATE.

- Масштабируемость и Шардирование:
В будущем финансовые данные могут расти быстрее, чем профили. Раздельные таблицы позволяют вынести wallets на отдельный диск или даже в отдельный инстанс БД (Sharding) без необходимости перекраивать всю авторизацию.

- Аудит и Безопасность:
Для финансовых таблиц часто требуются более строгие политики безопасности (Row Level Security) и детальное логирование (Triggers для истории). Проще настроить это на компактной таблице кошельков, чем на огромной таблице пользователей с кучей метаданных.
